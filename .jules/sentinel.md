## 2025-02-28 - Hardcoded API Key and Fail-Open Configuration
**Vulnerability:** The API server had a hardcoded `swarm_dev_key` fallback, allowing unauthorized access if the environment variable was missing. The frontend also hardcoded this key, leaking it to the client side.
**Learning:** Hardcoding credentials in fallback parameters circumvents environment-based security entirely, resulting in a fail-open posture when configuration issues occur. Similarly, putting static keys in client-side code completely invalidates their utility.
**Prevention:** Always implement a fail-closed architecture—if a required security token is missing from the environment, the server should fail to start or reject all requests with a 500 error. **Vulnerability:** Found f-string query in `get_active_swarms` in `src/messaging/db.py` directly interpolating the `timeout_seconds` variable into the query strings for a SQLite database operation instead of using parameterized arguments.
**Learning:** This is a risky pattern that can lead to SQL injection vulnerabilities, especially if input types change in the future or parameters are pulled from untrusted sources. The vulnerability existed because an f-string was a convenient way to build a SQL `datetime()` modifier dynamically.
**Prevention:** Always use parameterized SQL queries (e.g., `?` in SQLite) and handle dynamic components cleanly within the database's functions using proper string concatenation (`'-' || ? || ' seconds'`) rather than string interpolation in Python.
